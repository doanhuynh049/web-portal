src/lib/types.ts

Central type definitions — all other files derive their types from here. Three changes were made.

Modified

A — Theme union type extended to include "system"

Before
export type Theme = "dark" | "light";
After
export type Theme = "dark" | "light" | "system";

Adding "system" as a third option allows the app to follow the OS dark/light preference via window.matchMedia("(prefers-color-scheme: dark)"). Every component that references Theme now needs to handle three states instead of two.

New Field

B — UserProfile.avatarIcon optional emoji field

Before
export interface UserProfile {
  name: string;
  initial: string;
  avatarColor: string;
  role: string;
}
After
export interface UserProfile {
  name: string;
  initial: string;      // auto-derived from name
  avatarColor: string;
  role: string;
  avatarIcon?: string;  // NEW — emoji override e.g. "🚀"
}

The ? makes it optional so existing data (without the field) is valid. When avatarIcon is a non-empty string, components show the emoji; when empty/undefined they show the letter monogram. Stored as a separate DB column.

src/lib/db/schema.ts

Drizzle ORM schema — defines the PostgreSQL table structure. Adding avatarIcon to the profile requires a new column definition.

New Column

profileAvatarIcon added to portalSettings table

Before
  profileRole: text("profile_role").notNull().default("Developer"),
  version: integer("version").notNull().default(2),
After
  profileRole: text("profile_role").notNull().default("Software Developer"),
  profileAvatarIcon: text("profile_avatar_icon").default(""),  // NEW
  version: integer("version").notNull().default(2),

The column is declared without .notNull() so existing rows (that predate this column) remain valid. Drizzle ORM uses this schema to generate type-safe SQL queries — the column name in Postgres is profile_avatar_icon (snake_case) and the TypeScript property is profileAvatarIcon (camelCase).

src/lib/db/index.ts

Contains ensureTables() — the DDL runner that creates tables on first request. Existing tables need an ALTER TABLE to gain the new column.

New

ALTER TABLE ... ADD COLUMN IF NOT EXISTS migration

Added after the CREATE TABLE for portal_settings
await sqlHttp`
  ALTER TABLE portal_settings
  ADD COLUMN IF NOT EXISTS profile_avatar_icon TEXT DEFAULT ''
`;

Why ALTER TABLE? CREATE TABLE IF NOT EXISTS only runs once — if the table already exists, no new columns are added. For a production database that already has the portal_settings table, this ALTER TABLE adds the missing column. The IF NOT EXISTS clause makes it idempotent (safe to call multiple times). This pattern is used instead of a separate migration runner.

src/lib/store.ts

The dual-storage facade — all reads and writes to Neon pass through here. New fields must be mapped in both directions.

Modified

A — Read mapping: Neon row → TypeScript object

Before
theme: settingsRow.theme as "dark" | "light",
profile: {
  name: settingsRow.profileName,
  initial: settingsRow.profileInitial,
  avatarColor: settingsRow.profileAvatarColor,
  role: settingsRow.profileRole,
},
After
theme: settingsRow.theme as "dark" | "light" | "system",
profile: {
  name: settingsRow.profileName,
  initial: settingsRow.profileInitial,
  avatarColor: settingsRow.profileAvatarColor,
  role: settingsRow.profileRole,
  avatarIcon: settingsRow.profileAvatarIcon ?? "",  // NEW
},

The ?? (nullish coalescing) handles rows created before the column existed — they return null from Postgres, which becomes an empty string here. The theme cast now includes "system" so "system" stored in the DB is returned correctly instead of being widened to an incorrect type.

Modified

B — Write mapping: upsert includes profileAvatarIcon

Added to both INSERT values and ON CONFLICT DO UPDATE set
// In .values({...})
profileAvatarIcon: s.profile.avatarIcon ?? "",

// In .onConflictDoUpdate({ set: {...} })
profileAvatarIcon: drizzleSql`excluded.profile_avatar_icon`,

Why two places? The INSERT handles new rows; the ON CONFLICT DO UPDATE handles existing rows (upsert pattern). drizzleSql\`excluded.profile_avatar_icon\` is Drizzle ORM syntax that references the proposed INSERT value — meaning "use the value we just tried to insert". Without this, saving the profile would leave the column unchanged on update.

src/components/portal-app.tsx

The main client shell — all theme state lives here. Three changes were made to support system mode.

Modified

A — applyThemeClass reads OS preference for system mode

Before
function applyThemeClass(t: "dark" | "light") {
  const root = document.documentElement;
  if (t === "light") root.classList.add("light");
  else root.classList.remove("light");
}
After
function applyThemeClass(t: Theme) {
  const root = document.documentElement;
  const isDark =
    t === "system"
      ? window.matchMedia("(prefers-color-scheme: dark)").matches
      : t === "dark";
  if (isDark) root.classList.remove("light");
  else root.classList.add("light");
}

window.matchMedia("(prefers-color-scheme: dark)").matches returns true if the OS is in dark mode. This is a synchronous read — no async needed. The function now adds or removes the .light CSS class on <html> based on the resolved preference.

New

B — useEffect listens for OS preference changes

New effect — reacts to system theme switching
useEffect(() => {
  if (theme !== "system") return;
  const mq = window.matchMedia("(prefers-color-scheme: dark)");
  const handler = () => applyThemeClass("system");
  mq.addEventListener("change", handler);
  return () => mq.removeEventListener("change", handler);
}, [theme]);

When the user switches their OS from dark to light (or vice versa) while in System mode, this effect fires immediately and updates the CSS class. The cleanup function removes the listener when the effect re-runs (e.g. user switches to "dark" mode) — preventing memory leaks. The effect only activates when theme === "system"; for dark/light it returns early.

Modified

C — Toggle cycles through 3 states via lookup table

Before — binary toggle
const next: "dark" | "light" = theme === "dark" ? "light" : "dark";
After — 3-state cycle
const cycle: Record<Theme, Theme> = {
  dark: "light",
  light: "system",
  system: "dark",
};
const next = cycle[theme];

A Record lookup table maps each state to its successor. This is more readable and maintainable than chained ternaries. The cycle is: dark → light → system → dark.

A — deriveInitial() — monogram from display name

New utility function
function deriveInitial(name: string): string {
  const words = name.trim().split(/\s+/).filter(Boolean);
  if (words.length === 0) return "?";
  if (words.length === 1) return words[0][0].toUpperCase();
  return (words[0][0] + words[words.length - 1][0]).toUpperCase();
}

Replaces the manual "Avatar Initial (single character)" input. The monogram updates live as the user types their name. Only the first character is persisted in the DB (the initial field) for backward compatibility.
